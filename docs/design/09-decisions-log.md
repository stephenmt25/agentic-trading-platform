# Design Decisions Log (ADRs)

Architecture-Decision-Record style log for the design choices that need explicit justification. Each entry follows: **Decision · Context · Alternatives considered · Rationale · Consequences**. The enterprise-architect reading style: every decision is reviewable.

---

## ADR-001 — Three modes (HOT/COOL/CALM), not one design system

**Decision:** The platform operates in three distinct visual modes — HOT (cockpit), COOL (laboratory), CALM (office) — with mode-scoped semantic tokens. A single uniform density/typography is rejected.

**Context:** The user spends time across radically different tasks: split-second order entry, multi-minute debate review, multi-day profile authoring. A density and motion system that's right for one is wrong for another.

**Alternatives considered:**
1. Single uniform design system, density toggle only — rejected because token-level differences (motion budget, color saturation, agent-color permission) can't be expressed by density alone.
2. Five+ modes — rejected as over-engineering; three is the natural decomposition matching the user's mental modes.
3. Per-surface customization with no mode abstraction — rejected because it loses the cross-surface consistency the harness needs.

**Rationale:** The mode boundary is a *behavioral* boundary, not just a stylistic one. HOT mode disallows agent identity colors; CALM mode disallows them too. COOL mode permits them. Same with motion budgets. Encoding this in the token system makes "the rules" enforceable rather than aspirational.

**Consequences:**
- Components must be mode-aware (or mode-agnostic — only the canonical primitives are agnostic).
- Surfaces declare their mode via `data-mode="..."` on their root.
- Future additions (a tutorial mode? a presentation mode?) need explicit mode definitions, not ad-hoc styling.

## ADR-002 — Color is meaning; six semantic roles, not a hue palette

> **Amended by ADR-012 (2026-05-07).** The original "seven roles incl. agent.{n}" claim is amended: agent identities alias to `accent` and differentiate by glyph + label + position, leaving six chromatic roles platform-wide.

**Decision:** The color system has exactly seven semantic roles (bid, ask, neutral, accent, warn, danger, agent.{n}). No tokens for "purple" or "teal" exist outside agent identities.

**Context:** Trading platforms often grow ad-hoc color systems where green means "good," amber means "warning," and so on — but each color drifts toward decorative use over time, breaking the user's ability to read color as meaning.

**Alternatives considered:**
1. Full Tailwind palette as available — rejected; too many escape hatches.
2. Brand-color-led palette (single primary + variations) — rejected; can't express the bid/ask vs. danger semantic distinction.
3. Add more semantic roles (info, neutral-positive, etc.) — rejected; ambiguous categories invite misuse.

**Rationale:** Constraining the palette to seven roles forces every component to *justify* its color. If a component wants "blue" outside agent.ta, that's a code smell. The harness can refuse to generate any color reach outside this taxonomy.

**Consequences:**
- Brand expression must happen through type and layout, not color.
- Visual interest is generated by density and rhythm, not by chromatic variety.
- New surfaces that "feel similar" risk monotony — accept this as a feature, not a bug.

## ADR-003 — The canvas is the source of truth for trading profiles

**Decision:** Visual canvas authority is non-negotiable. `trading_profiles.pipeline_config` is the canvas state; `strategy_rules` is a build artifact. The Pipeline Canvas surface gets disproportionate design investment.

**Context:** Per CLAUDE.md §2C, the platform has already made this architectural decision. The design must reflect it.

**Alternatives considered:**
1. Code-first with canvas as visualization — rejected by architecture; the user-facing model is canvas.
2. Form-first with canvas as power-user feature — rejected; users author *primarily* in the canvas.
3. Canvas as one of many editors equally weighted — rejected; the canvas is privileged.

**Rationale:** This is an existing architectural decision (`libs/core/pipeline_compiler.py`). Design coherence requires the canvas to *feel* central — through its IA position (one of six top-level surfaces, the highest-degree node in the IA graph), through its keyboard shortcut allocation, through the "view canvas as run" feature in Backtesting.

**Consequences:**
- Removing the canvas surface is not feasible without architectural change.
- Backtests must archive the canvas snapshot at run time (P4 → mandatory feature).
- Users on mobile cannot edit canvases (legitimate gap; we accept it).

## ADR-004 — Agent reasoning is a first-class peer to order entry

**Decision:** Agent Observatory is a top-level surface with the same visual weight as Hot Trading. Agents are not background services hidden in logs.

**Context:** Praxis is an *agentic* platform. Hiding the reasoning surface would make the system feel like a black box, which is the opposite of trustworthy AI design. The platform's competitive identity is *visible reasoning*.

**Alternatives considered:**
1. Agent reasoning as a side-panel of Hot Trading — rejected; the side panel can't carry the depth needed for debate review and intervention.
2. Logs-only (no observatory surface) — rejected; logs ≠ a designed surface.
3. Per-agent surfaces (six agent dashboards) — rejected; the Observatory IS the cross-agent view.

**Rationale:** Per philosophy P3 ("the platform reasons visibly"). The Observatory is also where intervention happens — the user's only path to override an agent's output. Hiding this would make Praxis less safe, not just less expressive.

**Consequences:**
- Inventing UI vocabulary that doesn't exist in trading platforms (AgentTrace, ReasoningStream, DebatePanel, ConfidenceBar). Borrow conservatively from Cursor/Claude/Perplexity where possible.
- Higher coordination cost between frontend and the agent services (real-time WS, structured outputs).

## ADR-005 — Risk Control is its own surface, separate from Hot Trading

**Decision:** Despite both being HOT mode, Risk Control gets a dedicated surface with separate keyboard shortcuts and reliability requirements.

**Context:** Risk control failures are catastrophic. Bundling kill-switch UI into Hot Trading risks the kill switch being unavailable when Hot Trading itself is degraded.

**Alternatives considered:**
1. Risk panel embedded in Hot Trading — rejected; couples availability of two unequally-critical features.
2. Modal-only risk controls — rejected; modals are too easy to miss/dismiss; risk needs a persistent home.
3. Risk Control as a non-mode-aware "always on" overlay — rejected; would conflict with the mode system.

**Rationale:** `Cmd+Shift+K` works from anywhere. The Risk surface is the configural home; the shortcut is the access path. Separating concerns lets us treat Risk Control as a higher SLO tier.

**Consequences:**
- Two surfaces share the HOT mode visual language but Risk Control is slightly less dense (legibility over density).
- The kill-switch state pill in chrome is mandatory on every surface.

## ADR-006 — Mobile is read-only for HOT surfaces

**Decision:** Hot Trading mobile is monitor-only — view positions, view PnL, kill switch. No order entry on mobile.

**Context:** Mobile derivatives trading on retail apps is a known source of regret-trades — fat-finger mistakes from small touch targets on volatile instruments.

**Alternatives considered:**
1. Full mobile parity — rejected on safety grounds.
2. Tablet-only support — rejected as too restrictive.
3. Mobile order entry with extra confirmations — rejected; doesn't fully solve the touch-target problem.

**Rationale:** This is a values-driven UX choice, not a tech limitation. The platform's audience plans trades and executes them deliberately, mostly from a workstation. We design for that audience, not the impulse-trader.

**Consequences:**
- Hot Trading mobile is functionally limited; we accept the loss.
- Risk Control mobile keeps the kill switch — safety overrides convenience.
- Profiles, Tax, audit log surfaces ARE responsive (these are infrequent-but-valid mobile tasks).

## ADR-007 — No light mode at v1

**Decision:** Praxis ships dark-only at v1. Light theme is a future commitment, not a current scope.

**Context:** Designing for both modes well requires re-deriving every chart color, every agent identity, every elevation rule. Half-quality light mode is worse than no light mode.

**Alternatives considered:**
1. Auto-derived light mode (mathematical inversion) — rejected; produces ugly results in practice for finance UI.
2. Hand-tuned light mode at v1 — rejected; doubles design cost and cuts focus.
3. Light mode "for printing" only — rejected; users don't print this stuff.

**Rationale:** Trading platforms are predominantly dark-themed. Stripe, Linear, Notion all started single-mode. Light mode is a v2 conversation.

**Consequences:**
- All token names assume dark — `bg-canvas` is darker than `bg-panel`. Future light theme will need to re-do this layering.
- Skill issue: don't generate light-themed components in the meantime.

## ADR-008 — Numbers in tabular figures, always

**Decision:** All numeric values use `font-feature-settings: 'tnum' 1, 'cv11' 1`. The `.num-tabular` utility is applied broadly.

**Context:** Proportional digits in Inter cause numbers to dance during updates — `1` is narrower than `8`, so a price ticking from `42118` to `42218` shifts width and the eye loses its place.

**Rationale:** Tabular numerics is so basic to financial UI that it shouldn't be a thought; making it a token-level enforced default removes it from designer/developer cognition.

**Consequences:**
- Numbers in tables, badges, prices, sizes, percentages — all tabular.
- Body text *prose* should NOT use tabular (it makes prose ugly). Distinction enforced via token: `scale.body-dense` for tabular contexts, `scale.body` for prose.

## ADR-009 — Agentic UI vocabulary is custom, not borrowed wholesale

**Decision:** AgentTrace, ReasoningStream, ToolCall, DebatePanel, ConfidenceBar, AgentAvatar are bespoke components, not adapted from any single source.

**Context:** No existing trading platform has these surfaces. Some agentic dev tools (Cursor, Claude) have analogues but their structures are *conversation-shaped*, which is wrong for a trading-supervision context.

**Alternatives considered:**
1. Adopt Cursor's agent-log UI verbatim — rejected; too IDE-shaped.
2. Adopt Claude Desktop's chat-with-tools — rejected; chat-shape conflicts with the Hot mode and the supervisor's workflow.
3. Hide reasoning behind "expand details" affordances — rejected per ADR-004; reasoning is a peer surface.

**Rationale:** Trading is *not* a conversation. The user is supervising autonomous agents that emit structured outputs. The vocabulary should reflect that — *emissions*, not messages; *traces*, not turns; *debate panels*, not chat threads.

**Consequences:**
- Higher invention cost; less off-the-shelf code.
- The vocabulary is part of Praxis's identity — competitors will recognize it as Praxis-distinctive.

## ADR-010 — One brand-voice accent (indigo), used sparingly

**Decision:** Indigo #6366f1 is the platform's "voice." All interactive affordances, focus rings, primary actions use it. It is the only chromatic accent permitted outside the bid/ask/agent semantics.

**Context:** Without a single accent, the platform either has no voice or has many — both bad. With many accent colors, the bid/ask semantic gets diluted.

**Alternatives considered:**
1. Multi-color accent palette (different accents per surface) — rejected; competes with bid/ask semantic.
2. No accent (pure neutral + bid/ask only) — rejected; the platform feels voiceless and cannot signal "this action is the primary one."
3. A more saturated brand color (cyan, magenta) — rejected; pulls toward "consumer crypto" aesthetic.

**Rationale:** Indigo is cooler than fintech blue (less Stripe-shaped) and more considered than crypto purple (less Solana-shaped). It works in dark mode at multiple weights. Used sparingly, it telegraphs "decision."

**Consequences:**
- The "Run live" button, focus rings, primary buttons, decision-edge wires all share indigo.
- Brand identity is carried through layout and type, not through chromatic variety.

## ADR-011 — Typography: IBM Plex Sans + IBM Plex Mono

**Date:** 2026-05-07

**Decision:** The platform uses IBM Plex Sans for prose and UI, and IBM Plex Mono for tabular numerics, code, and IDs. The portfolio v1.0.0-draft proposal of Inter + JetBrains Mono is rejected.

**Context:** `frontend/app/layout.tsx` already loads Plex Sans and Plex Mono via `next/font/google`. Plex Mono has excellent tabular figures and a slightly engineered character that fits the "bridge of a research vessel" brief better than Inter's modern-fintech voice. Switching to Inter would be a font swap with no clear payoff, while switching the portfolio to match reality is a docs-only change.

**Alternatives considered:**
1. Switch to Inter as portfolio specifies — rejected; matches Linear/Stripe but loses the engineered character that motivated the original Plex choice.
2. Inter for prose + Plex Mono for tabular — rejected; complexity without clear payoff.

**Rationale:** The portfolio is the bug here, not the existing build. Plex's tabular numerics are first-rate; Plex Mono works as the dedicated `--font-tabular` rather than relying on a sans-with-tnum trick.

**Consequences:**
- `tokens.json` and `tokens.css`: `--font-sans` → IBM Plex Sans, `--font-mono` → IBM Plex Mono, `--font-tabular` aliased to `--font-mono` directly.
- `--font-feat-tabular` simplifies to `'tnum' 1` (the Inter-specific `cv11` and `ss01` features are dropped — Plex doesn't have them).
- `frontend/app/layout.tsx` is already correct; no change needed when the redesign branch consumes the tokens.
- Portfolio philosophy §4 vibe text needs no change; the moods listed are still accurate for Plex.

## ADR-012 — Agent identities: glyph + position, not chromatic

**Date:** 2026-05-07

**Amends:** ADR-002 (the "seven roles incl. agent.{n}" formulation).

**Decision:** All six agent identities (ta, regime, sentiment, slm, debate, analyst) alias to `color.accent.500` (indigo). Visual differentiation is carried by glyph, label, and column position only. There are six chromatic roles platform-wide (bid, ask, neutral, accent, warn, danger), and the 3-color discipline holds across all modes — including COOL.

**Context:** Existing `frontend/DESIGN-SYSTEM.md` commits to a maximum of three semantic hues. Portfolio v1.0.0-draft proposed a Cool-mode exception giving each agent its own perceptually-spaced hue. Ahead of any code, the user chose the stricter middle-ground: the 3-color discipline is universal; agents differentiate non-chromatically.

**Alternatives considered:**
1. **(a)** Strict — drop agent identity tokens entirely, no chromatic affordance whatsoever — rejected; some chromatic emphasis is needed when many agents are listed and accent is permitted everywhere already.
2. **(b)** Cool-mode chromatic exception with six perceptually-spaced hues — rejected; principled but pulls toward "consumer trading" aesthetic with too many hues, and dilutes the bid/ask semantic over time.
3. **(c)** Single accent for all agents, glyph + label + position differentiate — chosen.

**Rationale:** Closer to the "bridge of a research vessel" restraint and to Bloomberg-citation style than to Slack-bubble style. Forces multi-agent surfaces to lean on typography and structure (P3 reasoning visibility holds without leaning on color). The 3-color rule survives intact, which keeps the bid/ask semantic high-signal.

**Consequences:**
- `tokens.json` color.agent.{ta,regime,sentiment,slm,debate,analyst} all alias to `{color.accent.500}`.
- `tokens.css` `--color-agent-*` variables all `var(--color-accent-500)`.
- 01-design-philosophy.md §3 P2: rewritten — six chromatic roles; agent block called out as glyph-led.
- 04-component-specs/agentic.md will be updated in Phase 5 to reflect glyph-led differentiation (AgentAvatar, ConfidenceBar, DebatePanel must each declare their glyph contract).
- Multi-agent surfaces (Observatory, Canvas) need a strong icon set for the six agents — single-stroke 1.5px outlined; each glyph distinctive at 20×20.

## ADR-013 — Repo strategy: in-place rewrite + worktree

**Date:** 2026-05-07

**Decision:** The redesign happens in-place on `frontend/` on a parallel branch `redesign/frontend-v2`. A git worktree at `../aion-trading-redesign` lets the existing main UI (port 3000) and the redesign UI (port 3001) run concurrently for side-by-side review.

**Context:** Plan §0.3 surfaced three options; main risk is operational complexity vs. review tooling.

**Alternatives considered:**
1. **(b)** Parallel app at `frontend-v2/` shipping under `/v2/*` for a transition period — rejected; higher operational complexity without commensurate safety benefit for a pre-launch platform.
2. **(c)** Greenfield separate repo — rejected; overkill for a UI redesign with shared backend.

**Rationale:** `git diff main..redesign/frontend-v2 -- frontend/` is the highest-leverage review tool available. Worktrees solve "run both UIs locally" cleanly without conflicting with the existing `.claude/worktrees/` family used by Claude Code agents.

**Consequences:**
- Main is pushed to origin before branching to avoid divergence on a shared feature.
- Tag `pre-redesign-baseline` marks the cut point on main for rollback reference.
- The redesign worktree at `../aion-trading-redesign` has an independent `node_modules/` and runs Next dev on port 3001.
- Cutover (Phase 9) is a `--no-ff` merge from `redesign/frontend-v2` to `main`; the worktree is removed afterwards.

## ADR-014 — Accent hue: indigo #6366f1

**Date:** 2026-05-07

**Decision:** The platform "voice" is indigo `#6366f1` (`color.accent.500`). The existing `frontend/app/globals.css` `--primary: oklch(0.60 0.14 240)` (cool blue) is replaced on the redesign branch only.

**Context:** Portfolio specifies indigo at ADR-010. Existing `frontend/app/globals.css` ships with cool blue as `--primary`. Without resolving this before tokens land, the redesign branch would have two competing accents at the component level.

**Alternatives considered:**
1. Keep existing cool blue (`oklch(0.60 0.14 240)`) — rejected; closer to generic fintech blue, less considered than indigo.
2. A different brand color (cyan, magenta) — rejected per ADR-010 reasoning.

**Rationale:** ADR-010 already justifies indigo as the brand voice — cooler than fintech blue, more considered than crypto purple, neutral enough to not compete with bid/ask semantics. This ADR closes the open conflict against the existing `--primary`.

**Consequences:**
- No change required to `tokens.json` or `tokens.css` — they already specify indigo.
- On the redesign branch (Phase 4), `frontend/app/globals.css` `--primary` is replaced by `--color-accent-500: #6366f1` and downstream shadcn-shaped tokens (`--ring`, `--sidebar-primary`, etc.) realign to indigo.
- `next-themes` is removed from the redesign branch (no light mode v1, ADR-007).
- Main branch is unchanged; the cool-blue accent persists there until cutover.

## How to add a new ADR

When the design portfolio needs a new explicit decision:

1. Add a new section here with the same structure.
2. Cross-link from the relevant philosophy/component/surface doc.
3. If the decision overrides an earlier one, mark the earlier ADR as **Superseded by ADR-NNN** at its top.

ADRs are append-only after sign-off — they are the audit trail of why the design is what it is.

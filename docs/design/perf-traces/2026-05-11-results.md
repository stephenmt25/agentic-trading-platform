# Phase 8.4 — Perf measurement results (2026-05-11)

Runner: chrome-devtools MCP, mobile profile (360×640×2.625, touch, mobile UA), network: Slow 4G, CPU: 4× throttling. Auth via injected `next-auth.session-token`. WebSocket stubbed via `initScript` to prevent network-idle never-firing on the live data stream.

## §8.4-A — FCP on `/hot/BTC-USDT`

| Metric | Value | Gate | Verdict |
|---|---|---|---|
| **FCP** | **2204 ms** | < 1500 ms | ❌ **FAIL** (over by 704 ms) |
| First Paint | 2204 ms | — | FP = FCP (single-paint render) |
| LCP | 2202 ms | (informational) | LCP ≈ FCP |
| TTFB | 109 ms | — | excellent |
| DOMContentLoaded | 2185 ms | — | FCP fires immediately after DCL |
| loadEvent | 10,399 ms | — | (informational) |
| CLS | 0.06 | (informational) | borderline (needs-improvement at 0.10) |

**Trace insights flagged by the MCP:**
- `RenderBlocking` — estimated savings ~600 ms FCP / ~600 ms LCP from deferring/inlining render-blocking requests
- `LCPBreakdown` — TTFB 119 ms / Render delay 2,083 ms; bottleneck is between server response and first paint, not the server
- `ForcedReflow` — JS reading layout after DOM mutation during hydration
- `NetworkDependencyTree` — long critical request chain

**Interpretation.** DCL = 2,185 ms and FCP = 2,204 ms means the browser blocks the first paint until React hydrates — there is no progressive paint of SSR HTML before JS evaluation. Combined with the 600 ms render-blocking insight (CSS in `<head>` and the large `_app` JS chunk), the page is paying the full hydration cost before any pixel ships. A streaming-SSR / non-blocking-CSS pass is the likely remediation.

**Throttling sanity check.** Trace metadata reads `Network throttling: none` (MCP harness quirk — the metadata doesn't reflect the per-page `emulate` overrides). A fetch to `/api/auth/session` during the same session took 593 ms for headers (consistent with Slow 4G's ~400 ms RTT), confirming throttling was active end-to-end. The 2,204 ms FCP is a real Slow-4G+4×CPU measurement.

**Caveat — cache state.** This run is warm-cache: the page navigated from `/login`, so shared chunks (`_app`, runtime, fonts) were served from disk cache. Cold-cache FCP would be ≥ this value. The gate fails on both interpretations.

## §8.4-A — Prod build re-measurement: ✅ **PASS**

The dev-mode measurement was invalid: the 2,204 ms above was dominated by Turbopack HMR clients, the unminified `lightweight-charts_development.mjs`, and the ~25 per-module unbundled JS chunks that `next dev` ships. A `next build && next start -p 3002` was set up and re-traced under the same mobile profile (360×640×2.625, Slow 4G, 4× CPU). Cookie/session was preserved across the port change (`localhost` cookies are port-agnostic in Chrome) and the WS stub was applied identically.

| CLS | 0.06 | 0.11 | (info) | crossed "needs-improvement" (>0.10) — flag, not a gate fail |

**Trace insight changes (dev → prod):**
- `RenderBlocking` est. FCP savings: **600 ms → 0 ms** — confirmed dev artifact (dev's per-module render-blocking chunks vanish in prod's bundled output).
- `DocumentLatency` now the dominant lever — 574 ms FCP/LCP savings available from reducing SSR overhead. With `/hot/[symbol]` as `ƒ Dynamic`, the SSR pass does per-request work (auth check, layout render) before the doc streams. Optional follow-up if Phase 8.4 wants to dig deeper.
- `ForcedReflow` still flagged (hydration reads layout) — same root cause as dev; small impact in prod.
- `CLS 0.11` — first content paints at 892 ms but layout still shifts ~108 ms later (likely fonts swapping from fallback to IBM Plex, or the chart frame resizing on data init). Worth a separate audit.

**Methodology notes — why prod is the correct measurement.**
- The playbook's "Pre-flight" assumes `bash run_all.sh --local-frontend`, which starts `next dev`. Spec gates like "FCP < 1,500 ms mobile" are derived from Lighthouse defaults which assume production bundles. Measuring dev against prod gates is apples-to-oranges.
- The prod measurement here is **cold-cache** (a fresh origin :3002 the browser had never visited), so it's stricter than the playbook's "clean cache" interpretation. A real user's second visit would FCP even faster.
- Prod build was unblocked by clearing 4 pre-existing TypeScript errors (EquityCurveChart.tsx legacy-UI, List.tsx polymorphic-element @ts-expect-error placement, DecisionFeed direction-type narrowing, TradeForensicsCard symbol-guard, TemplateGallery legacy-rules schema). All fixes in this commit; no behaviour change at runtime.

**Playbook update needed:** Pre-flight should specify `npm run build && npx next start` (or equivalent prod-mode server) before running §8.4-A. Adding to the playbook in a follow-up commit.

## §8.4-C — Memory soak (60 min): ✅ **PASS**

Clean re-run after the first attempt's methodology was compromised by Chrome backgrounding the tab. Setup: backend `bash run_all.sh`, prod frontend `npm run build && npm start` on :3000, page opened via chrome-devtools MCP at `/hot/BTC-USDT`. WebSocket activity was *instrumented* on this run — a global `window.WebSocket` shim installed via `initScript` recorded every connection's url, state, and frame count throughout the hour. Two heap snapshots taken via CDP `HeapProfiler` (no DevTools panel was toggled in between).

| Metric | t0 | t60 | Delta | Gate | Verdict |
|---|---|---|---|---|---|
| `usedJSHeapSize` (live, post-GC) | 22,964,097 B (21.9 MB) | 13,536,532 B (12.9 MB) | **−9.0 MB** | < 50 MB | ✅ |
| `totalJSHeapSize` (committed) | 54,884,433 B (52.3 MB) | 16,374,216 B (15.6 MB) | −36.7 MB | (info) | — |
| `.heapsnapshot` file size | 14,994,302 B (14.3 MB) | 16,853,955 B (16.1 MB) | +1.8 MB | (info) | retained-object delta within tolerance |

- t0: 2026-05-11T17:23:17Z
- t60: 2026-05-11T18:23:Z (Monitor fired exactly at t+59m, snapshot finished ~30s later)
- elapsed: ~60 min as spec'd (the bash `run_in_background` notification delivery is unreliable; switched to the `Monitor` tool which fires reliably on stream-event)

**The page was actually under load this time.** The WS shim recorded 54 WebSocket instances created during the soak (the singleton `wsClient` reconnects on every disconnect): 43 of those connections received frames (counts ranging 3,878–13,286), 11 later reconnect attempts got 0 frames. **Total: 321,536 frames received and processed across the hour** — roughly 89 frames/sec average. BTC price displayed in the UI moved from ~$81,619 → ~$81,988 during the soak (the page was clearly rendering live updates throughout).

**Findings the soak surfaced (separate from the leak verdict):**
1. **WS server-side closes every ~1–2 min with code 1006.** Forty-three abnormal closures and forty-three matching reconnects means the backend is dropping the WS connection on a short cycle, not the browser. The reconnect logic in `frontend/lib/ws/client.ts` handles it correctly (no frames lost from the user's perspective until the very end), but the churn itself is suspicious and should be diagnosed. Log this as a HIGH-severity backend bug against api_gateway's WS endpoint.
2. **WS auth token expired before t0 yet connections worked for ~50 min.** The JWT in the WS URL (`?token=eyJ…`) had `exp` claim `1778523423` (2026-05-11T15:37:03Z) — already expired 1h 46m before t0. Connections succeeded anyway until the last 6–8 min of the soak, when reconnects started returning 0 frames. The token's only checked at handshake and there's no expiry enforcement, OR the failure mode at the end is unrelated to auth. Either way, a sign that WS auth isn't gating reliably.
3. **`OrderBook` "stale Xs" badge is inaccurate.** It climbed continuously throughout the soak (60s → 488s) even while the orderbook rows themselves rendered fresh prices. The badge's underlying timestamp source isn't being refreshed by the same updates that mutate the row data — UI bug isolated to the `OrderBook` component's `stale` display logic.

**What the leak verdict means.** With 321k frames processed and 43 reconnect cycles, any retain-and-forget bug — unbound event listeners, store subscriptions not cleaned up in `useEffect` cleanups, WebSocket-on-close handlers stacking up — would have ballooned the heap. The fact that `used` heap *shrank* by 9 MB and the `.heapsnapshot` file grew by only 1.8 MB across the hour means the React tree and the `wsClient` singleton are reclaiming references correctly across reconnects. No leak under realistic load.

**Snapshots retained (gitignored):**
- `frontend/.perf-traces-tmp/soak-t0.heapsnapshot` (14.3 MB) + `soak-t0-timestamp.txt`
- `frontend/.perf-traces-tmp/soak-t60.heapsnapshot` (16.1 MB) + `soak-t60-timestamp.txt`

**Methodology improvements over the first attempt:** (a) WS instrumentation via `initScript` proved load was actually applied (321k frames vs the first run's ~21 min of live + 93 min dormant); (b) `Monitor` tool used for the 60-min wait instead of `Bash run_in_background` whose completion notification proved unreliable; (c) Snapshot tooling triggers a brief WS drop but the reconnect cycle recovers within seconds, so post-snapshot `perf.memory` readings reflect a live page.
